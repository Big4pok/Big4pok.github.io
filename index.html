Опишите схему сетевой БД. Назовите отличие в организации данных иерархических и сетевых СУБД. Перечислите недостатки ранних СУБД.

  • Схема: Сетевая модель данных (Network Database Model):
    • В сетевой модели данные организованы в виде графа, где узлы представляют записи, а связи между ними — это отношения.
    • В отличие от иерархической модели, в сетевой модели каждая запись может быть связана с несколькими записями на разных уровнях, создавая структуру «многие ко многим».
    Основные элементы сетевой модели: запись, связь, схема данных
    • [Схема сетевой БД - media/image1.png]
  • Отличие:
    [Сравнение иерархической и сетевой моделей - media/image2.png]
    • Иерархическая модель:
      • Структура: Древовидная структура, где один корень и множество дочерних узлов.
      • Связи: Только «один ко многим».
      • Навигация: Доступ к данным осуществляется сверху вниз.
      • Ограничения: Сложно реализовать связи «многие ко многим».
    • Сетевая модель:
      • Структура: Графовая структура с произвольными связями между записями.
      • Связи: Поддерживает как «один ко многим», так и «многие ко многим».
      • Навигация: Переход осуществляется по связям между записями.
      • Гибкость: Легко реализует сложные связи между данными.
  • Недостатки ранних СУБД:
    • Отсутствие гибкости: Трудно изменять структуру данных без перепроектирования.
    • Сложность разработки: Требует детального понимания физической структуры данных.
    • Слабая поддержка стандартизированных языков: Работа с данными в основном осуществлялась через низкоуровневые команды и навигационные языки.
    • Отсутствие удобных интерфейсов для пользователей: Интерфейсы для работы с данными были сложными и неудобными.

Укажите, в чём заключается ограничение целостности сетевой и иерархической СУБД. Дайте определения основным понятиям реляционного подхода. Перечислите свойства, которыми должно обладать реляционное отношение.

  • Ограничения:
    • Иерархическая СУБД:
      • Ограниченная гибкость: Структура данных жестко фиксирована, связи «один ко многим» сложны для реализации других типов.
      • Удаление данных: Удаление родительской записи может повлечь потерю всех дочерних данных.
    • Сетевая СУБД:
      • Сложность поддержания целостности: Связи «многие ко многим» трудны для управления, и нарушение одной связи может привести к несогласованности данных.
      • Навигация по графу: Путаница в навигации между записями может нарушить целостность данных.
  • Определения:
    • Отношение – таблица данных, состоящая из строк и столбцов.
    • Атрибут - столбец в таблице, определяющий тип данных
    • Ключи:
      • Первичный ключ (Primary Key) – Уникально идентифицирует каждую строку в таблице.
      • Внешний ключ (Foreign Key) – Связывает данные между двумя таблицами и обеспечивает ссылочную целостность.
      • Составной ключ - Состоит из нескольких атрибутов для уникальной идентификации записи.
    • Домен - Множество допустимых значений для атрибута.
    • Транзакция - Последовательность операций с данными, которая выполняется как единое целое.
    • Схема базы данных - Описание структуры базы данных, включающее таблицы, атрибуты и связи между ними.
  • Свойства: единственность, отсутствие дублирования, связанность данных, независимость от порядка, интегрированность данных, целостность данных.
9.

Что такое первичный ключ отношения? Назовите назначения индексов. Для чего используется внешний ключ в таблице?

  • Первичный ключ - Это атрибут (или группа атрибутов), который уникально идентифицирует каждую строку в таблице. Он не может содержать NULL значений.
  • Назначения индексов: ускоряют поиск данных в таблице, повышают эффективность выполнения запросов.
  • Для чего?: Внешний ключ (Foreign Key): Это атрибут, который связывает одну таблицу с другой, обеспечивая ссылочную целостность. Внешний ключ в одной таблице указывает на первичный ключ другой таблицы.

Перечислите способы поддержания ссылочной целостности. Дайте определение реляционной алгебры. Дайте определение совместимости структур отношений.

  • Способы:
    1. Ограничение внешнего ключа (Foreign Key): Проверяет, чтобы значения в столбце соответствовали значениям в другом столбце.
    2. Каскадные операции:
      • CASCADE: Автоматически удаляет/обновляет связанные записи.
      • SET NULL: Устанавливает значение NULL в связанных записях.
      • RESTRICT/NO ACTION: Запрещает удаление/обновление связанных записей.
  • Определение: Реляционная алгебра — это математический язык для работы с реляционными базами данных, состоящий из набора операций, которые выполняются над таблицами.
  • Определение: Совместимость структур отношений означает, что два отношения могут быть объединены или связаны, если их атрибуты имеют одинаковые типы данных и соответствуют друг другу по порядку.

11. Операции реляционной алгебры
  • Основные операции:
    • Выбор (SELECT) - фильтрация строк по условию
    • Проекция (PROJECT) - выбор определенных столбцов
    • Объединение (UNION) - комбинация двух отношений
    • Пересечение (INTERSECTION) - общие строки двух отношений
    • Разность (DIFFERENCE) - строки из первого отношения, отсутствующие во втором
    • Соединение (JOIN) - объединение по общим атрибутам
  • Реляционное исчисление:
    • Декларативный язык описания запросов
    • Определяет какие данные выбрать, а не как их получить
    • Результат - подмножество данных, удовлетворяющее условию
  • Формы реляционного исчисления:
    • Кортежное - запросы выражаются через кортежи (строки)
    • Доменное - запросы выражаются через домены значений атрибутов
12. Реляционные отношения и кванторы
  • Реляционное отношение кортежей:
    • Таблица с уникальными строками (кортежами)
    • Каждый кортеж представляет отдельную запись
    • Порядок кортежей не имеет значения
    • Пример: таблица Employees с данными сотрудников
  • Реляционное отношение доменов:
    • Таблица с определенными доменами для каждого атрибута
    • Домен задает допустимые значения для столбца
    • Пример: столбец Age с доменом "целые числа от 18 до 65"
  • Кванторы:
    • ∃ (существования) - "существует хотя бы один"
    • ∀ (всеобщности) - "для всех", "каждый"
    • Используются в условиях запросов
13. Связь алгебры и исчисления
  • Связь реляционной алгебры и исчисления:
    • Оба являются формальными основами реляционных БД
    • Могут выражать одни и те же запросы
    • Алгебра - процедурный подход (как получить результат)
    • Исчисление - декларативный подход (что нужно получить)
  • Логическая эквивалентность:
    • Любой запрос в алгебре можно выразить в исчислении
    • Любой запрос в исчислении можно выразить в алгебре
    • Одинаковая выразительная мощность
  • Задачи проектирования БД:
    • Анализ требований предметной области
    • Создание концептуальной схемы (ER-диаграммы)
    • Логическое проектирование (реляционная модель)
    • Физическое проектирование (оптимизация хранения)
14. Методология проектирования
  • Фазы проектирования:
    1. Анализ требований - сбор информации о предметной области
    2. Концептуальное проектирование - создание ER-модели
    3. Логическое проектирование - преобразование в реляционную модель
    4. Физическое проектирование - оптимизация для конкретной СУБД
  • Подходы к концептуальной модели:
    • Объектно-ориентированный - акцент на объекты и их поведение
    • Сущностно-ориентированный - акцент на сущности и их атрибуты
  • Метод "сущность-связь" (ER):
    • Графическое представление структуры данных
    • Основные элементы:
      • Сущности (прямоугольники)
      • Атрибуты (овалы)
      • Связи (ромбы)
    • Типы связей: 1:1, 1:N, M:N
15. Нормализация отношений
  • Нормализация:
    • Процесс устранения избыточности данных
    • Устранение аномалий вставки, обновления, удаления
    • Разделение таблиц по функциональным зависимостям
  • Нормальные формы:
    Нормальная форма Условие
    1NF Все атрибуты атомарны, нет повторяющихся групп
    2NF Выполнена 1NF + нет частичных зависимостей от ключа
    3NF Выполнена 2NF + нет транзитивных зависимостей
  • Особенности реляционных СУБД:
    • Хранение данных в таблицах
    • Поддержка SQL (стандартизированного языка)
    • Использование индексов для ускорения поиска
    • Поддержка транзакций (ACID)
16. Внешняя память БД
  • Объекты внешней памяти:
    • Жесткие диски (HDD/SSD)
    • Магнитные ленты (для резервных копий)
    • Оптические диски (архивирование)
    • Облачные хранилища
  • Особенности хранения отношений:
    • Данные организованы в файлы и блоки
    • Один блок содержит несколько строк таблицы
    • Разделение данных по файлам для оптимизации
    • Использование индексов для быстрого поиска
  • Достоинства и недостатки:
    Достоинства Недостатки
    Большие объемы хранения Медленнее оперативной памяти
    Постоянное хранение Зависимость от скорости диска
    Экономичность Необходимость индексов для скорости
17. Индексы и журнализация
  • Назначение индексов:
    • Ускорение поиска данных в таблицах
    • Оптимизация выполнения запросов
    • Ускорение операций соединения (JOIN)
    • Обеспечение уникальности данных
  • Способы организации индексов:
    • B-деревья - сбалансированные деревья для диапазонных запросов
    • Хеш-индексы - для точного поиска по ключу
    • Битовые индексы - для столбцов с небольшим числом уникальных значений
    • Полнотекстовые индексы - для текстового поиска
  • Журнальная информация:
    • Используется для восстановления после сбоев
    • Содержит историю изменений данных
    • Позволяет откатывать незавершенные транзакции
    • Обеспечивает целостность данных
    • Используется для аудита изменений
18. Служебная информация и транзакции
  • Служебная информация:
    • Метаданные о структуре БД
    • Статистика распределения данных
    • Информация о блоках данных
    • Состояния транзакций
    • Параметры настройки СУБД
    • Журналы изменений
  • Транзакция:
    • Последовательность операций с БД
    • Выполняется как единое целое (принцип атомарности)
    • Либо полностью завершается (COMMIT)
    • Либо полностью отменяется (ROLLBACK)
    • Пример: перевод денег между счетами
  • Свойства транзакций (ACID):
    • Atomicity (Атомарность) - все или ничего
    • Consistency (Согласованность) - сохранение целостности
    • Isolation (Изолированность) - независимость транзакций
    • Durability (Долговечность) - сохранение результатов
19. Уровни изолированности
  • Уровни изолированности транзакций:
    Уровень Описание Проблемы
    Read uncommitted Чтение незафиксированных данных Грязное чтение, неповторяемое чтение, фантомы
    Read committed Только зафиксированные данные Неповторяемое чтение, фантомы
    Repeatable read Гарантирует повторяемость чтения Фантомы
    Serializable Полная изоляция Нет
  • Сериализация транзакций:
    • Порядок выполнения, эквивалентный последовательному
    • Обеспечивает корректность параллельного выполнения
    • Исключает взаимное влияние транзакций
  • Примеры конфликтов:
    • Потерянное обновление (две транзакции перезаписывают изменения)
    • Грязное чтение (чтение незафиксированных данных)
    • Неповторяемое чтение (разные значения при повторном чтении)
    • Фантомы (появление новых строк между чтениями)
20. Методы сериализации
  • Методы сериализации:
    • Блокировки:
      • Разделяемые (S) - для чтения
      • Эксклюзивные (X) - для записи
      • Двухфазный протокол блокировок
    • Временные метки:
      • Каждой транзакции присваивается метка
      • Выполнение согласно порядку меток
      • Откат при нарушении порядка
  • Синхронизационные захваты:
    • Механизм блокировки ресурсов
    • Предотвращает одновременное изменение одних данных
    • Типы: на уровне строк, страниц, таблиц
    • Может приводить к взаимоблокировкам (deadlock)
  • Распознавание тупиков:
    • Построение графа ожидания
    • Обнаружение циклов в графе
    • Использование таймаутов

25. Синтаксис SELECT
  • Базовый синтаксис: 
    SELECT [DISTINCT] column1, column2, ...
FROM table_name
[WHERE condition]
[GROUP BY column]
[HAVING group_condition]
[ORDER BY column [ASC|DESC]];
  • Примеры:
    • Простой запрос: 
      SELECT product_name, price 
FROM products 
WHERE category = 'Electronics';
    • С группировкой: 
      SELECT department, COUNT(*) as emp_count 
FROM employees 
GROUP BY department 
HAVING COUNT(*) > 5;
  • Операторы условий:
    • Сравнения: =, <>, <, >, <=, >=
    • Логические: AND, OR, NOT
    • Проверки: IN, BETWEEN, LIKE, IS NULL
    • Агрегатные: COUNT, SUM, AVG, MAX, MIN
26. Особенности SELECT
  • Использование в SELECT:
    • Числовые константы: 
      SELECT price * 1.2 as price_with_tax FROM products;
    • Символьные константы: 
      SELECT 'Product: ' || product_name as full_name FROM products;
    • Арифметические операции: 
      SELECT (salary * bonus) as total_income FROM employees;
  • Оператор GROUP BY:
    • Группирует строки по значениям столбцов
    • Используется с агрегатными функциями
    • Пример: 
      SELECT department, AVG(salary) as avg_salary
FROM employees
GROUP BY department;
  • Примеры сложных запросов:
    • С вычислениями: 
      SELECT product_id, quantity, price, 
       quantity * price as total
FROM order_items;
    • С группировкой: 
      SELECT EXTRACT(YEAR FROM order_date) as year,
       COUNT(*) as orders_count
FROM orders
GROUP BY EXTRACT(YEAR FROM order_date);
27. Агрегирующие функции
  • Агрегирующие функции:
    • COUNT() - подсчет строк 
      SELECT COUNT(*) FROM employees;
    • SUM() - сумма значений 
      SELECT SUM(salary) FROM employees;
    • AVG() - среднее значение 
      SELECT AVG(price) FROM products;
    • MAX()/MIN() - максимальное/минимальное 
      SELECT MAX(salary), MIN(salary) FROM employees;
  • Связанные подзапросы:
    • Подзапрос выполняется для каждой строки основного запроса
    • Имеет доступ к данным внешнего запроса
    • Пример: 
      SELECT e.name, e.salary, 
       (SELECT AVG(salary) FROM employees d 
        WHERE d.department = e.department) as avg_dept_salary
FROM employees e;
  • Алгоритм выполнения:
    1. Выполнение внешнего запроса до подзапроса
    2. Для каждой строки внешнего запроса:
      • Вычисление параметров подзапроса
      • Выполнение подзапроса
      • Возврат результата в основной запрос
    3. Формирование окончательного результата
28. Команды DML
  • Основные команды:
    • SELECT - выборка данных 
      SELECT * FROM customers WHERE country = 'USA';
    • INSERT - вставка новых строк 
      INSERT INTO products (name, price) VALUES ('Laptop', 999.99);
    • UPDATE - обновление данных 
      UPDATE employees SET salary = salary * 1.1 WHERE department = 'IT';
    • DELETE - удаление данных 
      DELETE FROM orders WHERE status = 'cancelled';
  • Оператор ORDER BY:
    • Сортировка результатов запроса
    • Может сортировать по нескольким столбцам
    • Примеры: 
      SELECT * FROM products ORDER BY price DESC;

SELECT name, salary FROM employees 
ORDER BY department ASC, salary DESC;
  • Примеры сложных запросов:
    • Сортировка с ограничением: 
      SELECT * FROM products 
ORDER BY stock_quantity ASC 
LIMIT 10; -- Товары с наименьшим запасом
    • Обновление с подзапросом: 
      UPDATE employees
SET bonus = 1000
WHERE id IN (SELECT employee_id FROM sales 
             WHERE amount > 10000);
29. Предложение HAVING
  • Назначение HAVING:
    • Фильтрация результатов группировки
    • Применяется после GROUP BY
    • Работает с агрегированными данными
    • Пример: 
      SELECT department, AVG(salary) as avg_salary
FROM employees
GROUP BY department
HAVING AVG(salary) > 50000;
  • Отличие от WHERE:
    WHERE HAVING
    Фильтрует строки до группировки Фильтрует группы после группировки
    Не работает с агрегатными функциями Работает с агрегатными функциями
    Выполняется раньше GROUP BY Выполняется после GROUP BY
  • Примеры связанных подзапросов:
    • С HAVING: 
      SELECT customer_id, COUNT(*) as order_count
FROM orders
GROUP BY customer_id
HAVING COUNT(*) > (SELECT AVG(order_count) 
                   FROM (SELECT COUNT(*) as order_count 
                         FROM orders GROUP BY customer_id));
    • Сложный пример: 
      SELECT department, 
       AVG(salary) as avg_salary,
       (SELECT MAX(salary) FROM employees e2 
        WHERE e2.department = e1.department) as max_salary
FROM employees e1
GROUP BY department
HAVING AVG(salary) > 40000
ORDER BY avg_salary DESC;
30. Подзапросы с INSERT и UNION
  • Подзапросы с INSERT:
    • Вставка данных из результата запроса
    • Пример: 
      INSERT INTO premium_customers
SELECT customer_id, name, email
FROM customers
WHERE total_purchases > 10000;
    • Вставка с вычислениями: 
      INSERT INTO sales_stats (product_id, avg_sale)
SELECT product_id, AVG(amount)
FROM sales
GROUP BY product_id;
  • Оператор UNION:
    • Объединяет результаты нескольких SELECT
    • Удаляет дубликаты (UNION ALL сохраняет)
    • Пример: 
      SELECT product_id FROM current_orders
UNION
SELECT product_id FROM archive_orders;
  • Примеры использования UNION:
    • Объединение данных из разных таблиц: 
      SELECT 'Employee' as type, name, salary FROM employees
UNION
SELECT 'Manager' as type, name, salary FROM managers
ORDER BY salary DESC;
    • С агрегацией: 
      SELECT 'Q1' as quarter, SUM(amount) as total FROM sales_q1
UNION
SELECT 'Q2', SUM(amount) FROM sales_q2
UNION
SELECT 'Q3', SUM(amount) FROM sales_q3
UNION
SELECT 'Q4', SUM(amount) FROM sales_q4;
31. Подзапросы с DELETE и EXISTS
  • Подзапросы с DELETE: 
    DELETE FROM products
WHERE category_id IN (
    SELECT category_id FROM categories 
    WHERE discontinued = 1
);

    Мой комментарий: Этот запрос удалит все товары из discontinued категорий. Подзапрос сначала находит ID таких категорий.

  • Оператор EXISTS: 
    SELECT * FROM employees e
WHERE EXISTS (
    SELECT 1 FROM sales 
    WHERE sales.employee_id = e.id AND amount > 10000
);

    Мой комментарий: EXISTS проверяет наличие хотя бы одной строки в подзапросе. Здесь мы находим сотрудников, совершивших продажи более 10000.

  • Пример NOT EXISTS: 
    SELECT * FROM customers c
WHERE NOT EXISTS (
    SELECT 1 FROM orders 
    WHERE orders.customer_id = c.id
);

    Мой комментарий: Находим клиентов, которые никогда не делали заказов. NOT EXISTS вернет true, если подзапрос не нашел ни одной строки.

32. Объединение таблиц
  • Что такое объединение:

    Механизм комбинирования данных из нескольких таблиц по связанным полям. Например, объединение таблиц заказов и клиентов по customer_id.

  • Виды объединений:
    • INNER JOIN - только совпадающие строки
    • LEFT JOIN - все строки левой таблицы + совпадения
    • RIGHT JOIN - все строки правой таблицы + совпадения
    • FULL JOIN - все строки обеих таблиц
    • CROSS JOIN - декартово произведение
  • Подзапросы с UPDATE: 
    UPDATE employees
SET salary = salary * 1.1
WHERE department_id IN (
    SELECT id FROM departments 
    WHERE profit > 1000000
);

    Мой комментарий: Повышаем зарплату на 10% сотрудникам прибыльных отделов. Подзапрос определяет какие отделы прибыльные.

33. Компоненты локальной сети и JOIN
  • Компоненты локальной сети:
    • Серверы (файловые, БД, приложений)
    • Рабочие станции (клиентские компьютеры)
    • Сетевые коммутаторы и маршрутизаторы
    • Кабельная инфраструктура (витая пара, оптоволокно)
    • Точки беспроводного доступа
    • Сетевое ПО и протоколы (TCP/IP)
  • Оператор JOIN: 
    SELECT o.order_id, c.name, o.amount
FROM orders o
JOIN customers c ON o.customer_id = c.id;

    Мой комментарий: JOIN объединяет таблицы orders и customers по customer_id, позволяя видеть имя клиента рядом с его заказом.

  • Пример сложного JOIN: 
    SELECT e.name, d.name as department, p.name as project
FROM employees e
JOIN departments d ON e.department_id = d.id
JOIN projects p ON e.project_id = p.id;

    Мой комментарий: Множественный JOIN связывает данные о сотрудниках, их отделах и проектах в одном запросе.

34. Внутренние и внешние соединения
  • Внутреннее соединение (INNER JOIN): 
    SELECT p.name, s.quantity
FROM products p
INNER JOIN stock s ON p.id = s.product_id;

    Мой комментарий: Возвращает только продукты, присутствующие на складе. Если продукта нет в stock, он не попадет в результат.

  • Внешнее соединение (OUTER JOIN): 
    SELECT p.name, s.quantity
FROM products p
LEFT OUTER JOIN stock s ON p.id = s.product_id;

    Мой комментарий: Возвращает ВСЕ продукты, даже если их нет на складе (quantity будет NULL). LEFT означает, что сохраняются все строки левой таблицы.

  • Разница на примере:

    Если в products 100 товаров, а в stock - 80:

    • INNER JOIN вернет 80 строк (только товары в наличии)
    • LEFT JOIN вернет 100 строк (все товары, с количеством или NULL)
    • RIGHT JOIN вернет 80 строк (как INNER, если нет лишних записей в stock)
    • FULL JOIN вернет 100+ строк (все записи из обеих таблиц)
35. Архитектура "клиент-сервер"
  • Описание архитектуры:

    Клиентское приложение отправляет запросы серверу БД, который обрабатывает их и возвращает результаты. Например, веб-сайт (клиент) запрашивает данные через PHP-скрипт (сервер приложений), который обращается к MySQL (сервер БД).

  • Команды DML:
    • SELECT - чтение данных
    • INSERT - добавление новых записей
    • UPDATE - изменение существующих данных
    • DELETE - удаление данных
  • Примеры взаимодействия: 
    -- Клиент отправляет:
SELECT * FROM products WHERE price < 100;

-- Сервер возвращает:
+----+-----------+-------+-------+
| id | name      | price | stock |
+----+-----------+-------+-------+
| 5  | Mouse     | 25    | 150   |
| 8  | Keyboard  | 45    | 80    |
+----+-----------+-------+-------+

    Мой комментарий: Клиент запрашивает недорогие товары, сервер БД обрабатывает запрос и возвращает только соответствующие данные.

36. Взаимодействие клиент-сервер
  • Взаимодействие:
    1. Клиент устанавливает соединение с сервером
    2. Отправляет SQL-запрос через сетевой протокол
    3. Сервер парсит и оптимизирует запрос
    4. Выполняет операции с данными
    5. Возвращает результат клиенту
    6. Закрывает соединение (или оставляет в пуле)
  • Функции клиента и сервера:
    Клиент Сервер
    Формирование запросов Обработка запросов
    Отображение данных Управление хранением
    Валидация ввода Обеспечение безопасности
  • InterBase:

    Реляционная СУБД с архитектурой клиент-сервер, поддерживающая транзакции, триггеры и хранимые процедуры. Особенности: малый размер, встраиваемость, низкие требования к ресурсам.

37. InterBase: типы данных и объекты
  • Типы данных:
    • Числовые: INTEGER, FLOAT, NUMERIC(p,s)
    • Строковые: CHAR(n), VARCHAR(n), BLOB
    • Дата/время: DATE, TIME, TIMESTAMP
    • Логические: BOOLEAN
    • Бинарные: BLOB SUB_TYPE 0
  • Объекты БД:
    • Таблицы (основные, временные)
    • Представления (VIEW)
    • Хранимые процедуры
    • Триггеры
    • Генераторы (последовательности)
    • Исключения
    • Домены (пользовательские типы)
  • Состав триггеров:
    • Событие - когда срабатывает (BEFORE/AFTER, INSERT/UPDATE/DELETE)
    • Условие - необязательное ограничение (WHEN)
    • Тело - выполняемый код на PSQL
    • Приоритет - порядок выполнения среди триггеров
38. Домены и триггеры в InterBase
  • Назначение доменов:

    Домены позволяют создавать пользовательские типы данных с предопределенными характеристиками для повторного использования в разных таблицах.

    CREATE DOMAIN POSITIVE_INT AS INTEGER
CHECK (VALUE > 0);

    Мой комментарий: Создали домен для положительных чисел, который можно использовать в любых таблицах, где нужны такие значения.

  • Синтаксис работы с доменами: 
    -- Создание
CREATE DOMAIN EMAIL AS VARCHAR(100)
CHECK (VALUE LIKE '%@%.%');

-- Использование
CREATE TABLE users (
    id INTEGER,
    email EMAIL  -- используем домен
);

-- Удаление
DROP DOMAIN EMAIL;
  • Синтаксис триггеров: 
    CREATE TRIGGER update_timestamp
ACTIVE BEFORE UPDATE ON products
AS
BEGIN
    NEW.updated_at = CURRENT_TIMESTAMP;
END

    Мой комментарий: Этот триггер автоматически обновляет поле updated_at при каждом изменении записи в таблице products.

39. Ограничения доменов и таблиц
  • Когда нельзя изменить/удалить домен:

    Если домен используется в определении таблиц, представлений или других объектов БД. Сначала нужно удалить или изменить все зависимые объекты.

    -- Пример ошибки:
DROP DOMAIN PHONE_NUMBER;
-- Ошибка: Domain PHONE_NUMBER is used in table CUSTOMERS
  • Назначение таблиц:

    Основные объекты для хранения данных в реляционных БД. Состоят из строк (записей) и столбцов (полей), связанных первичными и внешними ключами.

  • Синтаксис работы с таблицами: 
    -- Создание
CREATE TABLE employees (
    id INTEGER PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    salary NUMERIC(10,2) DEFAULT 0,
    department_id INTEGER REFERENCES departments(id)
);

-- Изменение
ALTER TABLE employees
ADD COLUMN email VARCHAR(100);

-- Удаление
DROP TABLE employees;
40. Ограничения и технология IBX
  • Ограничения таблиц:
    • NOT NULL - запрет NULL значений
    • UNIQUE - уникальность значений
    • PRIMARY KEY - первичный ключ
    • FOREIGN KEY - ссылочная целостность
    • CHECK - пользовательские условия
    CREATE TABLE products (
    id INTEGER PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    price NUMERIC(10,2) CHECK (price > 0),
    category_id INTEGER REFERENCES categories(id)
);
  • Команда CONSTRAINT:

    Позволяет именовать ограничения для последующего изменения или удаления:

    CREATE TABLE orders (
    id INTEGER,
    amount NUMERIC(10,2),
    CONSTRAINT pk_orders PRIMARY KEY (id),
    CONSTRAINT chk_amount CHECK (amount >= 0)

  • Технология IBX:

    InterBase Express - набор компонентов для работы с InterBase из Delphi/C++Builder. Включает TIBDatabase для соединения, TIBTransaction для управления транзакциями, TIBQuery для выполнения запросов.

41. Индексы и IBX
  • Назначение индексов:

    Ускоряют поиск данных в таблицах за счет создания дополнительных структур данных, подобных оглавлению книги. Особенно полезны для часто используемых условий WHERE и JOIN.

  • Синтаксис работы с индексами: 
    -- Создание
CREATE INDEX idx_customer_name ON customers(last_name, first_name);

-- Удаление
DROP INDEX idx_customer_name;

-- Уникальный индекс
CREATE UNIQUE INDEX idx_product_code ON products(code);

    Мой комментарий: Составной индекс по фамилии и имени ускорит поиск клиентов по этим полям.

  • Иерархия IBX:
    • TIBDatabase - управление соединением
    • TIBTransaction - управление транзакциями
    • TIBQuery - выполнение SQL-запросов
    • TIBSQL - оптимизированные запросы
    • TIBTable - доступ к таблицам
    • TIBStoredProc - работа с хранимыми процедурами
42. Исключения и компоненты IBX
  • Назначение исключений:

    Позволяют централизованно обрабатывать ошибки в триггерах и хранимых процедурах. Можно создавать собственные исключения для бизнес-логики.

    CREATE EXCEPTION negative_salary 'Salary cannot be negative';

CREATE TRIGGER check_salary FOR employees
BEFORE INSERT OR UPDATE AS
BEGIN
    IF (NEW.salary < 0) THEN
        EXCEPTION negative_salary;
END
  • Синтаксис исключений: 
    -- Создание
CREATE EXCEPTION invalid_email 'Invalid email format';

-- Изменение
ALTER EXCEPTION invalid_email 'The email address is not valid';

-- Удаление
DROP EXCEPTION invalid_email;
  • Компоненты IBX для подключения:
    • TIBDatabase - настройка параметров соединения
    • TIBTransaction - управление транзакциями
    • TIBSQLMonitor - мониторинг SQL-запросов
    • TIBEvents - обработка событий БД
43. Триггеры в БД
  • Определение триггера:

    Хранимый код, который автоматически выполняется при наступлении определенного события (INSERT, UPDATE, DELETE) в таблице.

  • Назначение триггеров:
    • Проверка данных перед записью
    • Автоматический расчет значений
    • Ведение журнала изменений
    • Реализация сложных ограничений целостности
    • Синхронизация связанных данных
  • Компоненты IBX для работы с данными:
    • TIBQuery - выполнение произвольных SQL-запросов
    • TIBDataSet - набор данных с возможностью обновления
    • TIBTable - доступ к таблице "как есть"
    • TIBUpdateSQL - управление обновлением данных
44. Хранимые процедуры
  • Определение:

    Хранимая процедура - это именованный набор SQL-операторов, хранящийся в БД и выполняемый на сервере.

  • Назначение:
    • Инкапсуляция сложной бизнес-логики
    • Повышение производительности (меньше сетевого трафика)
    • Усиление безопасности (контроль доступа)
    • Упрощение клиентского кода
    • Централизованное управление логикой
  • Преимущества:
    • Быстрее отдельных SQL-запросов
    • Меньше нагрузки на сеть
    • Повторное использование кода
    • Безопасность (нет инъекций SQL)
    • Согласованность данных
45. Виды хранимых процедур
  • Виды процедур:
    • Системные - встроенные в СУБД
    • Пользовательские - созданные разработчиками
    • Временные - существующие только во время сеанса
    • Расширенные - на языках вроде Java, C#
  • Операторы для работы: 
    -- Создание
CREATE PROCEDURE update_salary(employee_id INTEGER, increase NUMERIC)
AS
BEGIN
    UPDATE employees 
    SET salary = salary + increase
    WHERE id = employee_id;
END

-- Вызов
EXECUTE PROCEDURE update_salary(123, 500);
  • Компоненты IBX для информации:
    • TIBDatabaseInfo - информация о БД
    • TIBSQLMonitor - мониторинг SQL-запросов
    • TIBExtract - извлечение метаданных
46. Преимущества и недостатки IBX
  • Преимущества IBX:
    • Нативная работа с InterBase/Firebird
    • Высокая производительность
    • Полный набор компонентов
    • Поддержка всех функций СУБД
    • Хорошая документация
  • Недостатки IBX:
    • Только для Delphi/C++Builder
    • Нет кроссплатформенности
    • Сложность миграции на другие СУБД
    • Ограниченное сообщество

    Универсальный framework для доступа к SQL-СУБД. Плюсы: кроссплатформенность, поддержка многих СУБД. Минусы: меньшая производительность, ограниченная функциональность.

55. Трехзвенная архитектура
  • Функция добавления данных: 
    function addProduct(PDO $db, string $name, float $price): int 
{
    $stmt = $db->prepare("INSERT INTO products (name, price) VALUES (?, ?)");
    $stmt->execute([$name, $price]);
    return $db->lastInsertId();
}

// Использование
$productId = addProduct($db, 'New Product', 99.99);
  • Уровни архитектуры:
    1. Презентационный - пользовательский интерфейс

    3. Доступ к данным - работа с БД и хранилищами
  • Функции уровней:
    • Презентационный: отображение, ввод данных, валидация

    • Доступ к данным: CRUD-операции, транзакции
56. Многоуровневые приложения

  • Менеджер транзакций:

    Координирует выполнение транзакций, обеспечивая ACID-свойства. Состоит из:

    • Монитора - отслеживает состояние
    • Средств восстановления - обработка сбоев
    • Средств блокировок - управление доступом
  • Преимущества:
    • Масштабируемость - можно масштабировать каждый уровень отдельно
    • Гибкость - уровни можно модифицировать независимо
    • Надежность - изолированные сбои на одном уровне
